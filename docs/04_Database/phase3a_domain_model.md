## Part 1: Domain-Scoped Business Entities

This document defines the core domain model and business entities. Every entity listed follows the permanent scale (100k+ users), multi-tenant, RBAC, XAI, and security constraints of the platform.

---

### Identity Domain

#### 1. Tenant (Organization)
- **Purpose**: Represents a distinct business customer account (corporate/group tenant) or the default public retail tenant.
- **Responsibilities**: Manages billing isolation, custom domains, security configurations, and user membership scopes.
- **Owned By**: System
- **Relationships**: 
  - One-to-Many with `User`
  - One-to-Many with `Subscription Plan` (Instances)
  - One-to-Many with `Audit Log`
- **Lifecycle**: Created when a new organization registers. Suspended on payment failure. Soft-deleted during offboarding.
- **Security Considerations**: Tenant boundaries are cryptographically or logically isolated. Tenant-specific settings must be protected from access by other tenants.
- **Multi-Tenant Considerations**: The root boundary for all SaaS isolation. Every operational record in downstream domains belongs to a tenant.

#### 2. User
- **Purpose**: Represents an individual human account accessing the platform.
- **Responsibilities**: Manages login credentials, holds profile information, maps to tenant membership, and defines permission sets.
- **Owned By**: Tenant (except System Admins owned by System)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-Many with `Role`
  - One-to-One with `User Profile`
  - One-to-Many with `API Key`
  - One-to-Many with `Session`
- **Lifecycle**: Invited or registered -> Active -> Suspended (billing or policy) -> Archived (soft-deleted).
- **Security Considerations**: Passwords stored using Argon2id. MFA configurations (TOTP/WebAuthn) are bound directly to this entity.
- **Multi-Tenant Considerations**: Scoped strictly under one `Tenant` namespace. A user cannot view data belonging to other tenants.

#### 3. Role
- **Purpose**: Defines a logical group of access permissions.
- **Responsibilities**: Groups permissions to simplify access control (e.g., Admin, Analyst, Premium User, Free User).
- **Owned By**: System (Default roles) / Tenant (Custom roles)
- **Relationships**:
  - Many-to-Many with `User`
  - Many-to-Many with `Permission`
- **Lifecycle**: Static for system-defined roles; tenant-defined roles are created, modified, or retired on demand.
- **Security Considerations**: Hardened against permission escalation. Role changes must invalidate existing sessions.
- **Multi-Tenant Considerations**: Default roles are shared across tenants; custom roles are isolated to the tenant that defined them.

#### 4. Permission
- **Purpose**: The lowest granularity of access authorization.
- **Responsibilities**: Explicitly controls write/read/execute actions on system features (e.g., `backtest:execute`, `signals:read`).
- **Owned By**: System
- **Relationships**:
  - Many-to-Many with `Role`
- **Lifecycle**: Defined statically in code/configurations and instantiated during system updates.
- **Security Considerations**: Immutable at runtime to prevent runtime permission manipulation.
- **Multi-Tenant Considerations**: Global and tenant-agnostic, but mapped dynamically within tenant-scoped roles.

#### 5. User Profile
- **Purpose**: Stores non-security user details.
- **Responsibilities**: Holds billing addresses, display names, phone numbers, avatars, and regional localization settings.
- **Owned By**: User
- **Relationships**:
  - One-to-One with `User`
- **Lifecycle**: Created automatically upon User creation. Maintained and updated by the user. Deleted with the User.
- **Security Considerations**: Contains Personally Identifiable Information (PII). Subject to encryption at rest and DPDPA (India) compliance (right to erasure/correction).
- **Multi-Tenant Considerations**: Strictly scoped to the parent User, inheriting the tenant boundaries of the parent.

#### 6. API Key
- **Purpose**: Provides programmatic API access for advanced users or third-party integrations.
- **Responsibilities**: Authenticates scripts and API queries without using user passwords.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `User`
- **Lifecycle**: Generated by user -> Active -> Revoked or Expired -> Purged.
- **Security Considerations**: Stored as a secure hash (SHA-256) in the database. Raw key shown to the user only once. Strictly rate-limited.
- **Multi-Tenant Considerations**: Mapped to a user, thus executing requests strictly within that user's tenant boundaries.

#### 7. Session
- **Purpose**: Tracks active user authentication state.
- **Responsibilities**: Manages session duration, token blacklisting, and device telemetry.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `User`
- **Lifecycle**: Created on login -> Refreshed on activity -> Terminated on logout/expiry/revocation.
- **Security Considerations**: Session tokens (JWTs) are signed using RS256. Invalidated automatically on password changes or security alerts.
- **Multi-Tenant Considerations**: Restricts tokens to the tenant context extracted during login.

### Billing & Subscription Domain

#### 8. Subscription Plan
- **Purpose**: Defines feature limits, pricing tiers, and quota boundaries.
- **Responsibilities**: Enforces SaaS limits (e.g., watchlists allowed, alerts per day, ML model depth). Tiers: Free, Trial, Pro, Enterprise.
- **Owned By**: System
- **Relationships**:
  - One-to-Many with `Tenant` (active plans)
- **Lifecycle**: Managed by product administrators. Active plans cannot be deleted if referenced; they are archived instead.
- **Security Considerations**: Immutable by tenant users to prevent plan bypass.
- **Multi-Tenant Considerations**: Global catalog read by all tenants to evaluate feature permissions.

#### 9. Billing Invoice
- **Purpose**: Documents fee assessments and billing events.
- **Responsibilities**: Records line items, plan charges, taxes (GST in India), currency, payment terms, and PDF download states.
- **Owned By**: Tenant
- **Relationships**:
  - Many-to-One with `Tenant`
  - One-to-Many with `Payment Transaction`
- **Lifecycle**: Draft -> Sent -> Paid -> Voided / Refunded.
- **Security Considerations**: Read-only once finalized. Protected against cross-tenant document viewing.
- **Multi-Tenant Considerations**: Scoped per tenant; contains billing parameters specific to that tenant.

#### 10. Payment Transaction
- **Purpose**: Records funds capture.
- **Responsibilities**: Integrates with payment processors (Razorpay/Stripe), tracking statuses, transaction IDs, fees, and failure codes.
- **Owned By**: Tenant (linked to User actions)
- **Relationships**:
  - Many-to-One with `Billing Invoice`
- **Lifecycle**: Pending -> Processing -> Settled / Failed / Disputed / Refunded.
- **Security Considerations**: Strictly audit-logged. Never stores raw card data or processing secrets.
- **Multi-Tenant Considerations**: Bound strictly to the parent tenant's ledger.

### Portfolio Domain

#### 11. Portfolio
- **Purpose**: Holds stock, derivative, and asset holdings for user performance and risk calculations.
- **Responsibilities**: Tracks cumulative performance, cost basis, realized/unrealized P&L, and asset allocations.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - One-to-Many with `Holding`
  - One-to-Many with `Order`
- **Lifecycle**: Created by User -> Active -> Archived.
- **Security Considerations**: Access governed by RBAC (e.g., Free users limited to 1 portfolio, Pro to 5 portfolios).
- **Multi-Tenant Considerations**: Scoped to User and Tenant. No query can fetch portfolios across tenant domains.

#### 12. Holding
- **Purpose**: Tracks inventory of a specific symbol inside a portfolio.
- **Responsibilities**: Computes cost basis (FIFO method), holding periods (short-term vs. long-term for tax calculations), and current value.
- **Owned By**: User (via Portfolio)
- **Relationships**:
  - Many-to-One with `Portfolio`
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Initiated on buy order execution -> Incremented/Decremented on transaction -> Liquidated (quantity = 0).
- **Security Considerations**: Direct influence on user P&L; updates must be transaction-consistent.
- **Multi-Tenant Considerations**: Strictly scoped to the parent portfolio's tenant.

#### 13. Order
- **Purpose**: Represents a buy/sell trade request.
- **Responsibilities**: Tracks target quantity, price limits, execution status, slippage, and brokerage fees. Dual-use for Paper Trading and Live Trading (future).
- **Owned By**: User (via Portfolio)
- **Relationships**:
  - Many-to-One with `Portfolio`
  - Many-to-One with `Market Symbol`
  - One-to-Many with `Broker Order` (future)
- **Lifecycle**: Created -> Submitted -> Partially Filled -> Filled / Cancelled / Expired / Rejected.
- **Security Considerations**: Access restricted to the owning user. Input sanitization prevents manipulation of execution prices.
- **Multi-Tenant Considerations**: Inherits tenant context from the portfolio.

#### 14. Watchlist
- **Purpose**: Tracks user-selected symbols for direct monitoring and scoped alerts.
- **Responsibilities**: Groups symbols, applies custom alerts, and configures dashboard priorities.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - One-to-Many with `Watchlist Item`
- **Lifecycle**: Created -> Modified (add/remove items) -> Deleted.
- **Security Considerations**: Read/write access strictly limited to the owner.
- **Multi-Tenant Considerations**: Strictly scoped to the User and Tenant.

#### 15. Watchlist Item
- **Purpose**: The mapping entity between a watchlist and a financial symbol.
- **Responsibilities**: Represents a single watched instrument with optional user tags or target notification prices.
- **Owned By**: User (via Watchlist)
- **Relationships**:
  - Many-to-One with `Watchlist`
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Created on item add -> Deleted on item remove.
- **Security Considerations**: Prevents reference to unauthorized or delisted symbols.
- **Multi-Tenant Considerations**: Inherits parent tenant scope.

### Alerts & Notifications Domain

#### 16. Alert Rule
- **Purpose**: Defines triggers for specific market events.
- **Responsibilities**: Evaluates conditions (e.g., price crossing threshold, indicator targets, AI signal triggers).
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Draft -> Active -> Triggered (if one-shot) -> Paused -> Deleted.
- **Security Considerations**: Rate limits apply (e.g., prevents infinite loop alert generation). Validated against active user subscriptions.
- **Multi-Tenant Considerations**: Enforces limits on alert rule creation based on the tenant's subscription plan.

#### 17. Alert Event (Alert History)
- **Purpose**: Stores historical alerts generated by rules.
- **Responsibilities**: Records transaction times, trigger reasons, signal values, and delivery statuses.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - Many-to-One with `Alert Rule`
  - One-to-Many with `Notification Log`
- **Lifecycle**: Generated -> Read/Acknowledged -> Deleted / Archived after 90 days.
- **Security Considerations**: Access is restricted to the owning user.
- **Multi-Tenant Considerations**: Tenant-scoped partition guarantees that tenants cannot inspect other alerts.

#### 18. Notification Preference
- **Purpose**: Configures user alert delivery channels.
- **Responsibilities**: Maps specific alert types to channels (Telegram, WhatsApp, Email, Push).
- **Owned By**: User
- **Relationships**:
  - One-to-One with `User`
- **Lifecycle**: Created during user onboarding. Updated by the user. Deleted on user termination.
- **Security Considerations**: Stores channel-specific endpoints (e.g., encrypted webhook URLs, verified email/phone numbers).
- **Multi-Tenant Considerations**: Inherits user tenant boundary.

#### 19. Notification Log
- **Purpose**: Logs delivery attempts and statuses.
- **Responsibilities**: Records API responses, error messages, and retry traces for notification channels.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Alert Event`
- **Lifecycle**: Created on delivery attempt -> Success or Failed.
- **Security Considerations**: Suppresses PII (like phone numbers/emails) in log text.
- **Multi-Tenant Considerations**: Scoped to the parent user's tenant.

### Market Data Domain

#### 20. Exchange
- **Purpose**: Represents a financial market registry.
- **Responsibilities**: Manages trading sessions, calendars, holiday lists, and base timezone properties (e.g., NSE, BSE).
- **Owned By**: System
- **Relationships**:
  - One-to-Many with `Market Symbol`
- **Lifecycle**: Static configuration; updated during regulatory alterations.
- **Security Considerations**: Read-only for standard users.
- **Multi-Tenant Considerations**: Shared system metadata available across all tenants.

#### 21. Market Symbol
- **Purpose**: Represents a financial instrument (stock, index, option, future).
- **Responsibilities**: Holds ticker names, ISIN, lot sizes, margins, listing segments, and industry classifications.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Exchange`
  - One-to-Many with `Historical Price Data`
  - One-to-Many with `Live Market Data`
- **Lifecycle**: Synchronized daily with exchange masters. Active -> Suspended -> Delisted.
- **Security Considerations**: Read-only. Protects symbol mappings to prevent invalid trade placement.
- **Multi-Tenant Considerations**: Global system entity accessible to all tenants.

#### 22. Historical Price Data (OHLCV)
- **Purpose**: Holds time-series historical price records.
- **Responsibilities**: Stores Open, High, Low, Close, Volume, Open Interest (for derivatives), and VWAP across multiple granularities (1m to 1D).
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Written in real-time aggregation jobs or imported during backfills. Read indefinitely; column-compressed over time.
- **Security Considerations**: Read-only database access.
- **Multi-Tenant Considerations**: Globally queried across all tenant strategy engines and AI modules.

#### 23. Live Market Data (Ticks)
- **Purpose**: Represents real-time tick-by-tick records.
- **Responsibilities**: Captures last traded price (LTP), last traded quantity (LTQ), bid/ask sizes, and market depth.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Ephemeral. Captured in WebSocket memory pipelines, written to stream topics, and discarded or compacted into OHLCV.
- **Security Considerations**: Low-latency read paths must be protected from resource exhaustion attacks.
- **Multi-Tenant Considerations**: Streamed concurrently to all active subscriber WebSockets.

#### 24. Corporate Action
- **Purpose**: Tracks equity structure modifications (dividends, splits, bonuses, mergers).
- **Responsibilities**: Records adjustment ratios and ex-dates for historical price adjustments.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Announced -> Ex-Date -> Executed (triggers price adjustments) -> Completed.
- **Security Considerations**: High integrity required. Price adjustments directly impact portfolio calculations.
- **Multi-Tenant Considerations**: Global metadata applied universally to all tenant portfolios.

#### 25. Technical Indicator
- **Purpose**: Represents mathematical derivations of price/volume data.
- **Responsibilities**: Calculates metrics (RSI, EMA, Bollinger Bands, MACD, etc.) across multiple timeframes.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Re-computed on new candle finalization. Cached in Redis; archived to time-series tables.
- **Security Considerations**: High availability target. Failure blocks strategy engines.
- **Multi-Tenant Considerations**: Read by all tenant scanners.

#### 26. News
- **Purpose**: Captures financial news and announcements.
- **Responsibilities**: Records headlines, article text, publisher sources, and corporate filing transcripts.
- **Owned By**: System
- **Relationships**:
  - Many-to-Many with `Market Symbol`
  - One-to-One with `News Sentiment`
- **Lifecycle**: Polled/Ingested -> Active -> Archived after 1 year.
- **Security Considerations**: News databases are read-only for standard users.
- **Multi-Tenant Considerations**: Shared system data.

#### 27. News Sentiment
- **Purpose**: Derives numerical sentiment scores from news.
- **Responsibilities**: Houses sentiment ratings (-1.0 to 1.0) and model confidence scores for a given article-symbol combination.
- **Owned By**: System
- **Relationships**:
  - One-to-One with `News`
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Generated during news ingestion. Immutable.
- **Security Considerations**: Linked directly to AI pipeline feature vectors.
- **Multi-Tenant Considerations**: Shared input for tenant strategies and AI evaluations.

#### 28. Market Regime
- **Purpose**: Defines macro market states (bullish, bearish, sideways, highly volatile).
- **Responsibilities**: Classifies broad market dynamics using index averages and volatility indexes (VIX).
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Exchange`
- **Lifecycle**: Re-evaluated daily or hourly.
- **Security Considerations**: Read-only.
- **Multi-Tenant Considerations**: Global signal applied to risk models.

#### 29. Economic Event
- **Purpose**: Captures macroeconomic calendar events (GDP, CPI, interest rates).
- **Responsibilities**: Records release dates, actual figures, consensus expectations, and historical impacts.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Exchange`
- **Lifecycle**: Scheduled -> Actual data released -> Consolidated -> Archived.
- **Security Considerations**: High integrity required.
- **Multi-Tenant Considerations**: Global system entity.

### AI Engine Domain

#### 30. AI Feature
- **Purpose**: The structured data vector fed to AI models.
- **Responsibilities**: Tracks over 150 indicators, sentiment indexes, regimes, and fundamental metrics per symbol and timestamp.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Market Symbol`
  - Many-to-One with `Feature Store Version`
- **Lifecycle**: Computed in stream processing -> Written to Feature Store -> Read for inference -> Archived for training.
- **Security Considerations**: Model features are proprietary IP. Read paths are restricted.
- **Multi-Tenant Considerations**: Shared dataset utilized by all tenant inference calls.

#### 31. Model Registry
- **Purpose**: Catalog of all machine learning models.
- **Responsibilities**: Manages metadata, model tags, and lifecycle states (Champion, Challenger, Archived).
- **Owned By**: System
- **Relationships**:
  - One-to-Many with `AI Model`
- **Lifecycle**: Permanent database of model families.
- **Security Considerations**: Access restricted to ML Engineers and System Admins.
- **Multi-Tenant Considerations**: Global registry serving predictions to all tenants.

#### 32. AI Model
- **Purpose**: Represents a specific model family (e.g., Price Direction LSTM, Volatility GARCH-ML).
- **Responsibilities**: Houses model descriptions, feature list requirements, and performance baselines.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Model Registry`
  - One-to-Many with `Model Version`
- **Lifecycle**: Registered -> Active -> Retired.
- **Security Considerations**: Model parameters and weights are encrypted and stored securely in private buckets.
- **Multi-Tenant Considerations**: Shared infrastructure entity.

#### 33. Model Version
- **Purpose**: A specific trained iteration of a model family.
- **Responsibilities**: Links to physical artifact storage (S3), records hyperparameter configurations, and tracks model verification metrics.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `AI Model`
  - One-to-Many with `AI Prediction`
  - One-to-Many with `Model Monitoring`
  - One-to-Many with `Model Drift`
  - One-to-One with `Confidence Calibration`
  - Many-to-One with `Feature Store Version`
- **Lifecycle**: Draft (training) -> Evaluated -> Candidate (challenger) -> Production (champion) -> Deprecated.
- **Security Considerations**: Model weights are checked for file integrity (checksums).
- **Multi-Tenant Considerations**: Serves inference requests dynamically across all tenants.

#### 34. AI Prediction
- **Purpose**: The output of online inference.
- **Responsibilities**: Records buy/sell predictions, volatility forecasts, target values, and model confidence scores.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Model Version`
  - Many-to-One with `Market Symbol`
  - One-to-One with `SHAP Explanation`
  - One-to-Many with `Trade Signal`
- **Lifecycle**: Generated -> Written to stream -> Read by signal engines -> Archived after 14 days.
- **Security Considerations**: Integrity protected to prevent fraudulent trade signaling.
- **Multi-Tenant Considerations**: Shared prediction stream consumed by tenant-specific signal processors.

#### 35. SHAP Explanation
- **Purpose**: Power the explainable AI (XAI) feature.
- **Responsibilities**: Holds feature contributions and human-readable explanation texts.
- **Owned By**: System
- **Relationships**:
  - One-to-One with `AI Prediction`
  - Many-to-One with `AI Prompt Library`
- **Lifecycle**: Computed during inference -> Delivered with predictions -> Displayed on User interface.
- **Security Considerations**: Proprietary SHAP configuration data must not be exposed.
- **Multi-Tenant Considerations**: Shared explanation payload; displayed contextually under user watchlists/portfolios.

### AI Intelligence (Extended Domain Models)

#### 50. Trade Signal
- **Purpose**: Represents an actionable trade recommendation derived from the fusion of strategy rules and AI models.
- **Responsibilities**: Stores direction (BUY/SELL), entry parameters, stop-loss, profit targets, combined confidence score, rationale snippet, and state validation.
- **Owned By**: System (Global recommendations) / Tenant (Tenant-custom signal setup)
- **Relationships**:
  - Many-to-One with `Market Symbol`
  - Many-to-One with `AI Prediction`
  - One-to-One with `Signal Performance`
  - One-to-Many with `AI Feedback`
  - One-to-Many with `Alert Event`
- **Lifecycle**: Generated -> Published -> Active (until Target/SL hit or expired) -> Archived.
- **Security Considerations**: High-integrity signature validation prevents fraudulent signal insertion. Encryption keys are stored in AWS Secrets Manager.
- **Multi-Tenant Considerations**: Global signals are tenant-agnostic; tenant-customized signals are isolated under the tenant scope.

#### 51. Signal Performance
- **Purpose**: Performs post-trade quantitative evaluation of trade signals.
- **Responsibilities**: Tracks entry execution, maximum favorable excursion (MFE), maximum adverse excursion (MAE), and final outcomes (SL hit, Target hit, Expiry).
- **Owned By**: System
- **Relationships**:
  - One-to-One with `Trade Signal`
- **Lifecycle**: Created at signal completion -> Metrics processed -> Saved as immutable performance ledger.
- **Security Considerations**: Read-only after final metrics write. Used for public strategy verification.
- **Multi-Tenant Considerations**: Shared analysis logs available to all subscribers to ensure accuracy audit transparency.

#### 52. AI Feedback
- **Purpose**: Captures user feedback on signal quality and trade behavior to refine future training inputs.
- **Responsibilities**: Records user ratings, execution notes, qualitative commentary, and structured feedback metrics.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Trade Signal`
  - Many-to-One with `User`
- **Lifecycle**: Created -> Submitted -> Saved (PII scrubbed) -> Consolidated in training dataset.
- **Security Considerations**: PII scrubbed from comments before dataset compilation.
- **Multi-Tenant Considerations**: Feedback lists are isolated strictly by User and Tenant.

#### 53. Scanner
- **Purpose**: Represents a real-time screening filter configuration.
- **Responsibilities**: Holds filter specifications (indicator levels, price parameters, volume criteria, AI feature conditions) and symbol universe constraints.
- **Owned By**: User (Custom Scanners) / Tenant (Tenant Group Scanners) / System (Built-in Scanners)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - One-to-Many with `Scanner Result`
- **Lifecycle**: Draft -> Active -> Suspended -> Retired.
- **Security Considerations**: Scanners are rate-limited to avoid system resource starvation.
- **Multi-Tenant Considerations**: Isolated strictly under the respective Tenant domain.

#### 54. Scanner Result
- **Purpose**: The output match of a scanner execution.
- **Responsibilities**: Records matched symbols and matching indicator/feature values at a specific execution timestamp.
- **Owned By**: User (via active query cache)
- **Relationships**:
  - Many-to-One with `Scanner`
  - Many-to-One with `Market Symbol`
- **Lifecycle**: Ephemeral. Cached in Redis (TTL: 1 minute) -> Expired and replaced.
- **Security Considerations**: Read-only access constraints.
- **Multi-Tenant Considerations**: Inherits scanner context isolation.

#### 55. Feature Store Version
- **Purpose**: Tracks feature sets versioning.
- **Responsibilities**: Defines schema configurations, data type validation logic, and ingestion version flags.
- **Owned By**: System
- **Relationships**:
  - One-to-Many with `AI Feature`
  - One-to-Many with `Model Version`
- **Lifecycle**: Draft -> Active -> Deprecated.
- **Security Considerations**: Configurations are protected.
- **Multi-Tenant Considerations**: Global feature store definition.

#### 56. Model Monitoring
- **Purpose**: Tracks live performance metrics of deployed AI Models.
- **Responsibilities**: Records execution latency (P50/P90/P99), throughput, CPU/GPU utilization, and memory usage.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Model Version`
  - One-to-Many with `Model Drift`
- **Lifecycle**: Logged per inference batch -> Aggregated hourly -> Stored in time-series logs.
- **Security Considerations**: Accessible only to System Admins.
- **Multi-Tenant Considerations**: System-wide operational logs.

#### 57. Model Drift
- **Purpose**: Tracks changes in data distribution (data drift) or concept distribution (concept drift).
- **Responsibilities**: Records population stability index (PSI) scores, Wasserstein distance, and divergence values.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Model Version`
  - Many-to-One with `Model Monitoring`
- **Lifecycle**: Calculated daily -> Triggers warning state -> Retraining pipeline executed -> Resolved.
- **Security Considerations**: System Admin access only.
- **Multi-Tenant Considerations**: System-wide monitoring.

#### 58. Confidence Calibration
- **Purpose**: Calibrates prediction probabilities to match real-world accuracies.
- **Responsibilities**: Holds temperature scaling variables and isotonic regression weights.
- **Owned By**: System
- **Relationships**:
  - One-to-One with `Model Version`
- **Lifecycle**: Computed during validation -> Used dynamically during inference -> Updated.
- **Security Considerations**: Protected weights.
- **Multi-Tenant Considerations**: Global calibration matrix.

#### 59. AI Prompt Library
- **Purpose**: Central library for LLM templates used in generating explanations.
- **Responsibilities**: Stores system prompt templates, few-shot contexts, and parameters.
- **Owned By**: System (Global Templates) / Tenant (Enterprise Customized Templates)
- **Relationships**:
  - One-to-Many with `SHAP Explanation`
- **Lifecycle**: Draft -> Active -> Deprecated.
- **Security Considerations**: Sanitizes inputs to prevent prompt injection attacks.
- **Multi-Tenant Considerations**: Custom templates are isolated.

### Trading & Broker Domain

#### 36. Broker Integration
- **Purpose**: Configuration metadata for broker adapters.
- **Responsibilities**: Manages OAuth settings, token exchange links, base endpoints, and rate limits (e.g., Zerodha Kite).
- **Owned By**: System
- **Relationships**:
  - One-to-Many with `Broker Account`
- **Lifecycle**: Maintained by administrators. Created -> Updated during API updates -> Retired.
- **Security Considerations**: Integration keys are stored in high-security secrets managers.
- **Multi-Tenant Considerations**: Global configurations used across all tenant broker calls.

#### 37. Broker Account
- **Purpose**: Links a user to their live brokerage account.
- **Responsibilities**: Stores encrypted access tokens, refresh tokens, broker client IDs, and active connection statuses.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - Many-to-One with `Broker Integration`
  - One-to-Many with `Broker Order`
- **Lifecycle**: Bound -> Connected (active token) -> Disconnected -> Unlinked.
- **Security Considerations**: Tokens are double-encrypted (application AES-256 + Secrets Manager). Access checks enforce tenant isolation.
- **Multi-Tenant Considerations**: Isolated strictly under Tenant and User.

#### 38. Broker Order
- **Purpose**: Represents a live order placed with an external broker.
- **Responsibilities**: Maps internal trade requests to external broker execution statuses, matching quantities, prices, and error codes.
- **Owned By**: User (via Broker Account)
- **Relationships**:
  - Many-to-One with `Broker Account`
  - Many-to-One with `Order`
- **Lifecycle**: Placed -> Pending external match -> Executed / Cancelled / Rejected.
- **Security Considerations**: Requires strict synchronization. Read/write actions must pass RBAC audits.
- **Multi-Tenant Considerations**: Tenant-isolated.

---

### Backtesting & Strategy Domain

#### 39. Strategy
- **Purpose**: Defines trading logic configurations.
- **Responsibilities**: Holds parameters, rules, symbol lists, timeframes, and indicators used in strategy evaluations.
- **Owned By**: System (Default strategies) / User (Custom strategies)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - One-to-Many with `Strategy Run`
- **Lifecycle**: Draft -> Approved (for running) -> Retired.
- **Security Considerations**: User-defined strategies are treated as proprietary user IP.
- **Multi-Tenant Considerations**: Custom strategies are isolated to the tenant context.

#### 40. Strategy Run
- **Purpose**: The execution trace of a strategy.
- **Responsibilities**: Links strategy runs to symbol data and tracks state changes.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Strategy`
  - One-to-Many with `Order` (for paper trading/backtesting)
- **Lifecycle**: Initiated -> Processing -> Completed -> Failed.
- **Security Considerations**: Strictly rate-limited to avoid system resource starvation.
- **Multi-Tenant Considerations**: Scoped to Tenant.

#### 41. Backtesting Configuration & Result
- **Purpose**: Configuration and outcome metrics of historical backtests.
- **Responsibilities**: Records CAGR, Sharpe, Sortino ratios, max drawdowns, slippage ratios, execution logs, and output PDF reports.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
  - Many-to-One with `Strategy`
- **Lifecycle**: Created -> Queued -> Running -> Completed -> Archived / Deleted.
- **Security Considerations**: Large reports stored in S3 are protected using presigned, authenticated URL access.
- **Multi-Tenant Considerations**: Backtest execution queues are tenant-isolated and bounded by rate limits.

---

### Risk Management Domain

#### 42. Risk Rule
- **Purpose**: Constraints to prevent capital loss.
- **Responsibilities**: Evaluates rules (e.g., sector allocation limits, maximum VaR limits, absolute stop-losses).
- **Owned By**: System (Global) / Tenant (Tenant-wide) / User (Portfolio-wide)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `Portfolio`
  - One-to-Many with `Risk Event`
- **Lifecycle**: Defined -> Active -> Deactivated.
- **Security Considerations**: Read-only for standard users. Must not be bypassable during execution.
- **Multi-Tenant Considerations**: Global and tenant-specific rules evaluate within the tenant scope.

#### 43. Risk Event
- **Purpose**: Tracks risk limit breaches.
- **Responsibilities**: Records details of limit violations, including breach types, triggered positions, values, and mitigation steps taken.
- **Owned By**: User (via Portfolio)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `Portfolio`
  - Many-to-One with `Risk Rule`
- **Lifecycle**: Raised -> Evaluated -> Sent to notification -> Resolved.
- **Security Considerations**: Immutable logs to preserve audit trails.
- **Multi-Tenant Considerations**: Isolated strictly under Tenant.

---

### Infrastructure & Operations Domain

#### 44. Audit Log
- **Purpose**: Tracks all platform admin and data modification actions.
- **Responsibilities**: Records target actors, IP addresses, execution contexts, changes made, and timestamps.
- **Owned By**: Tenant (or System for global admins)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
- **Lifecycle**: Created (write-only) -> Retained for 1 year -> Archived.
- **Security Considerations**: Structurally immutable (append-only). Encrypted to prevent tampering.
- **Multi-Tenant Considerations**: Scoped to the respective Tenant's admin dashboard.

#### 45. System Setting
- **Purpose**: Platform operational configurations.
- **Responsibilities**: Manages system parameters, API switches, maintenance modes, and external endpoint targets.
- **Owned By**: System (with Tenant override options)
- **Relationships**:
  - Many-to-One with `Tenant` (optional override)
- **Lifecycle**: Permanent; modified by administrators.
- **Security Considerations**: Access limited to System Administrators.
- **Multi-Tenant Considerations**: Global defaults apply unless overridden within a tenant scope.

#### 46. Feature Flag
- **Purpose**: Controls feature access and rollouts.
- **Responsibilities**: Manages access to platform features based on subscription plans, user groups, or beta tester pools.
- **Owned By**: System
- **Relationships**:
  - Many-to-One with `Subscription Plan`
- **Lifecycle**: Created -> In Testing -> Rollout -> Retired.
- **Security Considerations**: Evaluated on every request at the API Gateway to prevent unauthorized access.
- **Multi-Tenant Considerations**: Can be turned on/off for specific tenants or subscription tiers.

#### 47. Job
- **Purpose**: Tracks background asynchronous task executions.
- **Responsibilities**: Records celery task details, arguments, execution times, worker assignments, and statuses.
- **Owned By**: System (with Tenant ownership for user-initiated tasks)
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
- **Lifecycle**: Queued -> Active -> Success / Failed.
- **Security Considerations**: Task arguments must not log credentials or PII.
- **Multi-Tenant Considerations**: Tenant queues are isolated to prevent cross-tenant resource starvation.

#### 48. Scheduler Task
- **Purpose**: Manages recurring execution schedules.
- **Responsibilities**: Configures schedules (cron configurations) for tasks like data cleanup, report compilation, and cash checks.
- **Owned By**: System
- **Relationships**:
  - One-to-Many with `Job`
- **Lifecycle**: Active -> Paused -> Retired.
- **Security Considerations**: Modifications limited to System Admins.
- **Multi-Tenant Considerations**: Shared system infrastructure.

#### 49. Webhook Subscription
- **Purpose**: Sends real-time alerts to third-party endpoints.
- **Responsibilities**: Manages user-configured webhook target URLs, authentication headers, retry policies, and active states.
- **Owned By**: User
- **Relationships**:
  - Many-to-One with `Tenant`
  - Many-to-One with `User`
- **Lifecycle**: Created -> Active -> Suspended (on consecutive delivery failures) -> Deleted.
- **Security Considerations**: Outbound payloads are signed using HMAC-SHA256 to allow verifying authenticity.
- **Multi-Tenant Considerations**: Strictly scoped to the parent tenant.

### 1. Identity Domain
- **Entities**: `Tenant`, `User`, `Role`, `Permission`, `User Profile`, `Session`, `API Key`
- **Scope**: Access control, data isolation, and user profile management.

### 2. Billing & Subscription Domain
- **Entities**: `Subscription Plan`, `Billing Invoice`, `Payment Transaction`
- **Scope**: SaaS feature gating, invoicing, and subscription payments.

### 3. Market Data Domain
- **Entities**: `Exchange`, `Market Symbol`, `Historical Price Data`, `Live Market Data`, `Corporate Action`, `Technical Indicator`, `News`, `News Sentiment`, `Market Regime`, `Economic Event`
- **Scope**: Live and historical market data pipelines, news aggregation, and indicator computation.

### 4. AI Domain
- **Entities**: `AI Feature`, `Model Registry`, `AI Model`, `Model Version`, `AI Prediction`, `SHAP Explanation`, `Feature Store Version`, `Model Monitoring`, `Model Drift`, `Confidence Calibration`, `AI Prompt Library`
- **Scope**: Model lifecycle, feature extraction, predictions, SHAP explainability, system logging, and drift calibration.

### 5. Portfolio Domain
- **Entities**: `Portfolio`, `Holding`
- **Scope**: User asset tracking, cost basis, realized/unrealized P&L calculations, and allocations.

### 6. Trading Domain
- **Entities**: `Order`, `Broker Integration`, `Broker Account`, `Broker Order`
- **Scope**: Pluggable broker execution and paper trading order matching.

### 7. Backtesting & Strategy Domain
- **Entities**: `Strategy`, `Strategy Run`, `Backtesting Configuration & Result`, `Trade Signal`, `Signal Performance`, `AI Feedback`, `Scanner`, `Scanner Result`
- **Scope**: Strategy parameters, executions, backtest evaluations, and trade recommendations.

### 8. Alerts & Notification Domain
- **Entities**: `Alert Rule`, `Alert Event`, `Notification Preference`, `Notification Log`
- **Scope**: CEP alert evaluation, user watchlists, preferences, and delivery logs.

### 9. Risk Management Domain
- **Entities**: `Risk Rule`, `Risk Event`
- **Scope**: Portfolio risk monitoring, stop-loss triggers, and concentration rules.

### 10. Infrastructure & Operations Domain
- **Entities**: `Audit Log`, `System Setting`, `Feature Flag`, `Job`, `Scheduler Task`, `Webhook Subscription`
- **Scope**: System auditing, background task execution, task scheduling, and feature flags.
